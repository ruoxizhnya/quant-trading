### 3.2 核心发现

#### 发现 1: Report 文件的共同特征 ⚠️

| 特征 | 具体表现 | 影响 |
|------|---------|------|
| **快照性质** | 均为特定时间点的操作记录 | 信息随时间推移而衰减 |
| **一次性产出** | 创建后极少更新 | 无法反映后续变化 |
| **遗留问题过时** | 9 个遗留问题/风险项中 5 个状态不明或未解决 | 降低报告的可信度 |
| **相互关联** | MIGRATION_REPORT → FINAL_VERIFICATION_REPORT 存在包含关系 | 造成信息冗余 |
| **审计导向** | 主要服务于合规而非日常开发 | 使用频率低 |

#### 发现 2: 当前文档体系的问题 🔴

| 问题 | 具体表现 | 严重度 |
|------|---------|--------|
| **Report 膨胀** | 4 个 report 占用 ~34.4KB, 1044 行 | 🟠 High |
| **信息噪音** | 新开发者面对 28 个文档（含 4 个 report）难以定位关键信息 | 🟠 High |
| **维护负担** | 需要判断哪些 report 仍有效、哪些已过时 | 🟡 Medium |
| **缺乏生命周期管理** | 无明确的创建/更新/归档/销毁流程 | 🔴 Critical |
| **与 ADR 体系脱节** | 重要决策记录散落在 report 中，未纳入 ADR 统一管理 | 🔴 Critical |

#### 发现 3: Report vs ADR 的对比分析

| 维度 | Report (当前实践) | ADR (推荐实践) | 优势方 |
|------|------------------|----------------|--------|
| **格式标准化** | ❌ 各 report 格式不统一 | ✅ 严格遵循模板 | **ADR** |
| **状态追踪** | ❌ 无状态概念（只有"完成"）| ✅ Proposed→Accepted→Deprecated→Superseded | **ADR** |
| **不可变性** | ❌ 可被随意编辑 | ✅ 历史版本不可变，只追加修正 | **ADR** |
| **可检索性** | ❌ 分散在多个文件中 | ✅ 统一编号体系 + 索引表 | **ADR** |
| **机器可读** | ⚠️ 部分结构化 | ✅ YAML front matter + 标准章节 | **ADR** |
| **决策透明度** | ⚠️ 隐含在操作记录中 | ✅ Context→Decision→Consequences 显式结构 | **ADR** |
| **生命周期管理** | ❌ 无明确策略 | ✅ 归档/ supersede 机制清晰 | **ADR** |
| **适用场景** | ✅ 适合复杂操作的详细记录 | ✅ 适合重要决策的点状记录 | **各有侧重** |

## 五、基于 ADR 的长效文档管理机制设计

### 5.1 设计理念

**核心理念**: **将 "Report 思维" 转换为 "ADR 思维"**

| Report 思维 (当前) | ADR 思维 (目标) |
|-------------------|-----------------|
| "我完成了这项工作，写个总结报告" | "我做了一个重要决策，记录决策本身" |
| 关注 **过程** (Process) | 关注 **决策** (Decision) |
| 一次性产出，容易过时 | 不可变记录 + 状态流转 |
| 格式自由，难以机器解析 | 标准格式，结构化数据 |
| 分散在各处，难以检索 | 统一编号，集中索引 |

### 5.2 ADR 分类体系扩展

在现有 10 条 ADR (adr-001 至 adr-010) 的基础上，新增 **Operational ADR** (O-ADR) 子类：

#### 当前 ADR 分类

| 类别 | 编号范围 | 用途 | 数量 |
|------|---------|------|------|
| **Architectural ADR** (A-ADR) | adr-001 ~ adr-010 | 系统架构层面的重大技术决策 | 10 |
| **Operational ADR** (O-ADR) | odr-001 ~ odr-??? | 文档治理、流程优化、工具选型等运营决策 | 0 (待建) |

#### O-ADR 模板设计

```markdown
# ODR-[编号]: [简短标题]

> **Status**: [Proposed/Accepted/Completed/Deprecated]
> **Date**: [YYYY-MM-DD]
> **Category**: [Cleanup/Audit/Migration/Tooling/Process]
> **Related ADRs**: [adr-xxx, odr-yyy] (如有)
> **Supersedes**: [odr-zzz] (如替代旧决策)

## Context (背景)
[为什么要做这件事？遇到了什么问题？触发条件是什么？]

## Decision (决策)
[做了什么决定？选择了什么方案？为什么选择这个方案？]

## Consequences (后果)
[这个决策带来了什么影响？（正面+负面）]

## Artifacts (产出物)
[本次操作产出的文件/代码/配置清单]

## Metrics (度量指标)
[可量化的成果数据（如：文档数减少X%、耗时Y小时等）]

## Lessons Learned (经验教训)
[如果重来一次，会有什么不同的做法？]

---
_ODR created: [YYYY-MM-DD] by [Author]_
_ODR last updated: [YYYY-MM-DD] (status change only)_
```

### 5.3 Report → ADR 映射方案

将现有的 4 个 Report 转换为对应的 O-ADR：

| 原 Report | 对应 O-ADR | 核心决策点 | 状态 |
|-----------|-----------|-----------|------|
| CLEANUP_REPORT.md | **odr-001: Document Cleanup Operation** | 决定删除 4 个过时文档，目标从 26→22 | Completed |
| DOC_AUDIT_REPORT.md | **odr-002: Design Document Audit Framework** | 决定建立四维度审计框架（设计/一致性/测试/质量）| Completed |
| MIGRATION_REPORT.md | **odr-003: AGENTS.md Adoption Decision** | 决定采用 AGENTS.md 行业标准替代分散配置 | Completed |
| FINAL_VERIFICATION_REPORT.md | **odr-004: Verification Standard Definition** | 决定建立四项验证标准（一致性/无缺失/无额外/零影响）| Completed |

**转换原则**:
1. **提取决策**: 从 Report 中提取核心 "Decision"（做了什么决定）
2. **简化内容**: 删除详细的操作日志，保留 Context→Decision→Consequences 主线
3. **标准化格式**: 采用 O-DR 模板重写
4. **保留链接**: 在 O-DR 中添加指向原始 Report 归档位置的链接（如有需要）

### 5.4 长效维护机制

#### 机制 1: 触发式 O-DR 创建

当发生以下操作时，**必须**创建对应 O-DR：

| 触发事件 | O-DR 类别 | 创建时限 | 责任人 |
|---------|----------|---------|--------|
| 文档清理/瘦身操作 | Cleanup | 操作完成后 24h 内 | 执行者 |
| 设计文档审计/审查 | Audit | 审计完成后 48h 内 | 审计者 |
| 文档架构迁移/重构 | Migration | 迁移完成后 72h 内 | 架构师 |
| 开发工具/流程变更 | Tooling/Process | 变更发布前 | 技术负责人 |
| 重大 Bug 修复导致文档更新 | Process | 修复完成后 48h 内 | 修复者 |

#### 机制 2: O-DR 状态生命周期

```
Proposed (提议)
    ↓ [团队评审通过]
Accepted (采纳)
    ↓ [执行完成]
Completed (完成)
    ↓ [被新决策替代 或 12个月无活动]
Deprecated (废弃)
    ↓ [明确标注替代者]
Superseded (被取代)
```

**状态转换规则**:
- 只允许向前流转（不可回退）
- 每次状态变更必须追加记录（不可修改历史）
- Superseded 必须明确指向替代者的 O-DR 编号

#### 机制 3: 定期复审机制

| 复审类型 | 频率 | 执行人 | 范围 |
|---------|------|--------|------|
| **季度复审** | 每 3 个月 | Document Owner | 所有 Active 状态的 O-DR |
| **年度归档** | 每 12 个月 | Archive Admin | Completed 状态超过 6 个月的 O-DR |
| **按需复审** | 触发时 | Any Contributor | 特定 O-DR 被引用或质疑时 |

**复审检查清单**:
- [ ] 该 O-DR 描述的决策是否仍然有效？
- [ ] 是否有新的 O-DR 已经替代或推翻了这个决策？
- [ ] Consequences 中预测的影响是否已经发生？
- [ ] Lessons Learned 是否仍然适用？
- [ ] 是否需要更新 Status 或添加 Supersedes 链接？

#### 机制 4: 索引维护规则

**ADR.md (主索引)** 更新规范:
- 新增 O-DR 时：24h 内更新 ADR.md 索引表
- O-DR 状态变更时：即时更新 ADR.md 中的 Status 列
- O-DR 被 Superseded 时：在索引表中添加 ⚠️ 标记和替代者链接

**AGENTS.md Document Index** 更新规范:
- 仅保留 **Active** 和 **Recently Completed** (< 3个月) 的 O-DR 引用
- 归档的 O-DR 仅在 ADR.md 索引表中可见
